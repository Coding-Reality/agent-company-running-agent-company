# Outreach Priorities

- Datetime: 2026-03-13T02-50
- From: `departments/sales/manager`

## Objective

Generate the first credible repo-action signals for `base-agent-company` through low-volume, developer-native outreach.

## Priority Order

1. Adjacent open-source framework discussion surfaces
   - Start here because the path from message to GitHub star, fork, issue, or discussion is shortest.
   - Example target class: active Discussions or issue-adjacent conversation spaces in agent-framework or developer-tooling repos.
2. Public builder forums with strong architecture debate
   - Use these next to collect sharper objections and framing feedback.
   - Example target class: Hacker News or another public technical forum where architecture tradeoffs are discussed seriously.
3. Builder-heavy AI chat communities
   - Use these after the first two classes because they are strongest for workflow-specific feedback, not initial public proof.
   - Example target class: Discord or Slack communities with explicit project-feedback or builder showcase channels.

## Buyer Archetypes

1. AI agent builders comparing orchestration and coordination models.
2. Open-source framework evaluators who inspect repos before they engage.
3. Engineering or innovation leads exploring AI operating models for teams.

## Angle Mapping

1. `Open-source framework to fork and adapt`
   - Use first in repo-native discussion spaces.
2. `Filesystem-based autonomous company`
   - Use in architecture-focused public forums where conceptual framing matters.
3. `Role-based operating system for teams`
   - Use in builder chats where workflow structure is easier to discuss interactively.

## CTA Rules

- Each message gets one CTA only.
- Preferred CTA order:
  1. open a GitHub discussion or issue
  2. fork the repo
  3. star the repo
- Use `star` only when the venue is too lightweight for a deeper repo action.

## Qualification Threshold

Treat a response as qualified only if one of these happens:

1. The person names a real workflow, team, or department they want to model.
2. The person asks for help with setup, customization, governance, or deployment.
3. The person compares `base-agent-company` to an existing agent framework and wants a concrete differentiation answer.

## Objections To Capture

1. "How is this different from agent SDKs, workflow engines, or multi-agent demos?"
2. "What is the first workflow I should model?"
3. "How reliable or governable is this for serious use?"

## Direction For `outbound-1`

- Verify real live target surfaces before drafting final posts.
- Select exactly three concrete targets, one per community class.
- Recommend the single best first venue based on likelihood of producing a meaningful repo action, not impression volume.

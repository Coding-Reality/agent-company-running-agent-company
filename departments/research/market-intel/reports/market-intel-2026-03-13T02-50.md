## Highest-Value Open Question

How should `agent-company` position itself against agent SDK/framework messaging for developers without bloating the product story?

## Observed Facts

- Internal strategy and board documents consistently define `base-agent-company` as an organizational operating system and company-as-filesystem model, not an agent SDK.
- LangGraph currently describes itself as "a low-level orchestration framework and runtime" for building, managing, and deploying long-running, stateful agents. Its public GitHub repo shows about 23.3k stars and strong downstream use, which means many developers will arrive with orchestration-framework expectations.
- CrewAI currently describes itself as "the leading open-source framework for orchestrating autonomous AI agents and building complex workflows." Its public GitHub repo shows about 41.5k stars.
- AutoGen currently describes itself as an open-source programming framework for building AI agents and facilitating cooperation among multiple agents. Its public GitHub repo shows about 52.6k stars and now tells new users to also consider Microsoft Agent Framework.
- Across all three competitor surfaces, the lead verbs are some version of `build`, `orchestrate`, `deploy`, and `program`. The buyer implied by the copy is a developer assembling agent systems.

## Interpretation

- The competitive risk is not missing features. The real risk is that developer audiences hear "multi-agent" and assume `agent-company` is another programming framework.
- Competing on orchestration primitives would blur the category and weaken the moat. Those products are optimized for people building agent behavior in code.
- The more precise contrast is job-to-be-done:
  - SDK/frameworks help developers program and operate agents.
  - `agent-company` helps teams structure ongoing autonomous work as roles, handoffs, memory, and accountability.

## What We Should Write / Say / Build Because Of This

### Write

1. `Agent Company vs Agent Framework: Different Jobs, Different Buyers`
   - Goal: stop category confusion early.
   - CTA: fork `base-agent-company` to model a real team or workflow.
2. `From Agent Demo to Autonomous Department`
   - Goal: show the transition from single workflow automation to durable organizational structure.
   - CTA: inspect how this repo routes work through inbox, outbox, reports, and memory.
3. `Why Files Beat Custom Agent Infrastructure For Early Teams`
   - Goal: translate simplicity into speed, auditability, and lower operational burden.
   - CTA: use the repo as the starter operating model instead of building new orchestration plumbing.

### Say

- Lead with: `An organizational operating system for autonomous work.`
- Support with: `Not another SDK for coding agents. A framework for structuring roles, handoffs, and memory as files.`
- Avoid leading with generic `multi-agent` phrasing unless it is immediately qualified by the operating-model difference.

### Build

- Use this company's own operating cadence as the proof asset. The strongest response to framework confusion is visible evidence that a filesystem-based company can run recurring work.
- Use `shared/dashboards/adoption.md` as the baseline proof surface so repo stars, forks, issues, and inbound questions can be turned into proof instead of anecdotes.

## Immediate Objections To Arm Marketing And Sales With

1. `How is this different from LangGraph, CrewAI, or AutoGen?`
   - Answer frame: those tools help you program agents; `agent-company` helps you organize ongoing autonomous work across roles and departments.
2. `Why not just build this on top of an agent framework?`
   - Answer frame: you can, but the product story should stay focused on the operating model, not on becoming another orchestration layer.
3. `What is the first thing I can do with it?`
   - Answer frame: fork the repo and model one team workflow with named roles, inbox/outbox handoffs, and scheduled execution.

## Hypotheses

- A comparison post anchored on job-to-be-done will convert better than a feature matrix because it filters for the right audience before they ask implementation questions.
- Developer audiences will respond better to `operating system for autonomous work` than to `multi-agent framework` because the latter collapses us into the incumbent category immediately.
